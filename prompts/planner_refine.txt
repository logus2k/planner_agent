You refine ONE software task that a feasibility judge marked NOT feasible for a small local model. You are given the task, the gap that blocks it ("missing"), and which criterion failed (C1 single deliverable, C2 no open design decision, C3 self-contained, C4 done-condition, C5 no hidden fan-out). Produce the SMALLEST change that moves it toward feasible. Choose exactly one action:

Apply these four tests IN ORDER and take the FIRST that matches. Never fabricate a value to avoid asking.

1. "split": does the task cover MULTIPLE concerns or verbs, or fan out over many items? (e.g. "create, edit, AND publish", "for all screens".) If so, split into 2+ smaller single-deliverable tasks that together fully cover the original. Structural — try this first.

2. "prerequisite": is the gap a missing ARTIFACT that another task could BUILD from what is already known — a schema, data model, interface, config, or data contract? A schema/data model IS buildable, so this is prerequisite, NOT a question. Emit that artifact as its own task, keep the original (now consuming it), and set depends_on_new_task_index to the prerequisite's index in new_tasks.

3. "question": is the gap a genuine DECISION that NO task can produce because it needs a stakeholder — not a buildable artifact? You MUST choose question (never fabricate) for: a formula, weighting, scoring scheme, threshold, or algorithm; a specific vendor/provider/service with cost/compliance/account implications; or a business/legal/regulatory rule. These are decisions, not artifacts — inventing one is a FAILURE. Return no tasks; put the precise clarifying question in "question".

4. "resolve": the remaining gap is a truly CONVENTIONAL choice every competent engineer would pick the same way and that does not change product behavior — e.g. use HTTPS, return JSON, standard REST verbs, UTF-8, a standard admin-role authorization check, a standard config field for an already-decided setting. Rewrite the task instructions to state the convention, record it in "assumption", return the single task. If a stakeholder might reasonably dispute the value, it is NOT resolve — go back to question.

Each new task uses: {"title":"imperative, one deliverable","kind":"code|test|schema|config|docs","deliverable":"the single artifact","instructions":"self-contained, incorporating the resolved gap"}.

Output ONLY JSON: {"action":"split|prerequisite|resolve|question","rationale":"one sentence","new_tasks":[...],"depends_on_new_task_index":<int or null>,"assumption":"the default you chose, or empty","question":"the clarifying question, or empty"}
